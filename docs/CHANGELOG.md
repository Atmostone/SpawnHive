## 2026-05-23

- **E-04 — Behavioral Testing Layer (POC).** LLM-as-judge ненадёжна (смещения, разброс); objective execution-based проверки дают детерминированный сигнал. Новый evaluator `objective` в рубрике запускает статический анализатор над кодовыми артефактами задачи и складывает один measurement 0–10 в тот же профиль E-02 (Phase 1, `SPAWNHIVE_EVAL_BACKLOG.md` §E-04, Linear SPA-8).
  - **Движок `app/quality/objective.py`:** `evaluate_objective_dimension` с двумя пробами (`probe`): **lint** (`ruff check --output-format=json`) и **types** (`mypy --ignore-missing-imports`). Каждая проба берёт `.py`-файлы из `task.result_files`, тянет их из MinIO во временную папку, гоняет инструмент через `asyncio` subprocess с таймаутом, парсит вывод и считает score = `10 × (1 − min(findings_per_100_loc, 10)/10)` (0 находок = 10). Результат мемоизируется по content-hash артефактов. Контракт как у `_judge_dimension`/`reference`: никогда не бросает; нет `.py` → `status:"skipped"`; нет инструмента / таймаут / непарсящийся вывод → `status:"error"`. LLM не используется.
  - **Интеграция в `evaluate_task_quality` (judge.py):** `objective`-измерения скорятся в том же `asyncio.gather`, что judge/reference (изолированно); scored objective участвует в `weighted_score`/`gate` наравне с judge; `skipped`/`error` исключены. `human` остаётся единственным `deferred`-типом.
  - **API:** `DimensionBody.probe` (валидируется: при `evaluator="objective"` ∈ `{lint,types}`, по умолчанию `lint`; при прочих evaluator — обнуляется). Поле живёт в `dimensions` JSONB — **миграция не нужна**.
  - **Frontend:** в редакторе рубрик evaluator `objective` готов (без «(soon)») + select probe (lint/types); в `QualityRadarChart` probe показан в подписи измерения, tooltip `skipped` обобщён. `npm run build` зелёный.
  - **Зависимости:** в `requirements.txt` добавлены `ruff` и `mypy` (образ `api` пересобран).
  - **Тесты:** `tests/unit/test_objective.py` (маппинг score, реальные прогоны ruff/mypy, контракт skipped/scored/error, кеш, интеграция в профиль + weighted/gate) и `tests/integration/test_objective_eval.py` (персист/обнуление/дефолт `probe`, on-demand scored/skipped с monkeypatch-артефактом). Обновлён E-02-тест `test_rubric.py` (заглушка deferred переключена с `objective` на `human`). Полный backend-сьют: **236 passed**.
  - **Сознательные решения (POC-объём, согласован с пользователем):** **только статический анализ Python** (ruff/mypy) **in-process** — инструмент *парсит* код, не исполняет; **запуск тестов** (pytest/jest) исполняет недоверенный код агента и требует docker-изоляции → отдельная follow-up задача; web (Lighthouse/axe), text- и data-пробы, а также plugin-формат «новая проба = YAML+image» — отложены. - **E-03 — Reference-based Judge.** Для задач с известным эталонным ответом сравнение с эталоном — самый дешёвый и надёжный outcome-тип. Новый evaluator `reference` в рубрике сравнивает результат агента с золотым ответом задачи и складывает один скор 0–10 в тот же профиль E-02 (Phase 1, `SPAWNHIVE_EVAL_BACKLOG.md` §E-03, Linear SPA-7).
  - **Новое поле `tasks.reference_answer`** (TEXT NULL, миграция `d3e4f5a6b7c8`) — опциональный эталон, задаётся при создании задачи. Прокидывается через `TaskCreate`/`TaskUpdate`/`task_to_dict`.
  - **Движок `app/quality/reference.py`:** `evaluate_reference_dimension` с четырьмя режимами (`reference_mode`): **pointwise** (LLM-судья оценивает результат против эталона, та же модель `quality_judge` + fallback orchestrator, что в E-02), **exact** (10/0 после нормализации), **fuzzy** (`difflib.SequenceMatcher` ratio × 10, без новых зависимостей), **semantic** (косинус эмбеддингов через настроенный embedding-провайдер, считается in-process для двух текстов — без Qdrant). Контракт как у `_judge_dimension`: никогда не бросает; нет эталона → `status:"skipped"`, ошибка → `status:"error"`.
  - **Интеграция в `evaluate_task_quality` (judge.py):** `reference`-измерения скорятся в том же `asyncio.gather`, что и `judge` (изолированно — сбой одного не блокирует остальные); scored reference участвует в `weighted_score`/`gate` наравне с judge; `skipped` исключён из обоих. Pointwise-токены/стоимость идут в общие `judge_*` счётчики. `PROFILE_SCHEMA_VERSION` → 2. Триггеры не менялись — поток через существующие on-demand-кнопку и scheduler-джоб.
  - **API:** `DimensionBody.reference_mode` (валидируется: при `evaluator="reference"` ∈ `{pointwise,exact,fuzzy,semantic}`, по умолчанию `pointwise`; при прочих evaluator — обнуляется). `POST/PATCH /api/tasks` принимают `reference_answer`.
  - **Frontend:** опциональное поле «Reference answer» в форме создания задачи; в редакторе рубрик evaluator `reference` (готов, без «(soon)») + select режима; в `QualityRadarChart` режим показан в подписи измерения и добавлен счётчик `skipped`; в `TaskDetail` эталон выводится отдельным блоком. `npm run build` зелёный.
  - **Тесты:** `tests/unit/test_reference.py` (матчеры, контракт skipped/scored/error, интеграция в профиль + weighted/gate) и `tests/integration/test_reference_eval.py` (round-trip `reference_answer`, персист `reference_mode`, on-demand scored/skipped). Прогон reference+quality сьютов: **32 passed**; smoke API: **22 passed**.
  - **Сознательные решения:** **pairwise (A vs B vs reference) отложен** — требует второго кандидата-результата, которого в одной задаче нет (приедет с E-11/E-21); acceptance §1 выполнен на 2 из 3 режимов. Эталон берётся из поля задачи, а не из бенчмарка (E-23 ещё не сделан). Никаких хардкод-имён моделей (CLAUDE.md).
  - **Docs:** `docs/data-model.md` (`tasks.reference_answer` + `reference_mode` в измерении + миграция + schema_version 2), `docs/architecture.md` (подсекция Reference-based Judge + компонент `reference.py`), `docs/api.md` (`reference_answer` в task API + `reference_mode` в рубрике), `docs/overview.md` (строка E-03).

- **E-02 — Multi-dimensional Quality Rubric Engine.** Качество результата перестаёт быть одним числом (dimension collapse): рубрика оценивает результат по нескольким независимым измерениям (0–10) → **профиль качества** (вектор), который пишется в слот `quality_records.quality_profile` (E-01) и рисуется radar-чартом. Ядро outcome-стороны Phase 0 (`SPAWNHIVE_EVAL_BACKLOG.md` §E-02, Linear SPA-6).
  - **Новая таблица `rubrics`** (миграция `c2d3e4f5a6b7`): workspace-scoped; `dimensions` JSONB (`{key, name, description, evaluator, weight, threshold, critical}`), `applies_to` (тег типа задачи), `is_default`. 5 встроенных рубрик (Analytical Report / Code / Content / Design / Data Analysis) сидятся в default workspace (`seed_default_rubrics`) и клонируются новому ws при регистрации. Та же миграция добавляет `templates.rubric_id` (FK SET NULL) и `workspaces.quality_judge_model_id` (FK SET NULL).
  - **Judge-модель — отдельный системный слот `quality_judge`** (рядом с orchestrator/chat/memory_extractor; добавлен в `SystemModelKind`, `/api/workspaces/me/system-models`, клон при регистрации). **Fallback на orchestrator-модель**, если не задан — единственный consumer с fallback (eval некритичен, деградирует в «skipped», а не в ошибку).
  - **Движок `app/quality/{rubric,judge}.py`:** `resolve_rubric_for_task` (приоритет: `template.rubric_id` → тег `applies_to` ∩ `template.tags` → `is_default` → none); `evaluate_task_quality` — по одному LLM-вызову на judge-измерение через `asyncio.gather` с независимым try/except (один сбой не блокирует остальные → `status:"error"`), objective/human → `status:"deferred"`. Профиль несёт `weighted_score`, `gate{passed, failed_dimensions}`, judge model/tokens/cost. Блоб MinIO неизменяем — пишется только Postgres-колонка.
  - **Триггеры:** on-demand `POST /api/quality/records/{task_id}/evaluate` (owner/admin, перезапускает оценку) **+** scheduler-джоб `quality_judge_evaluate` (interval 600s, берёт `done`-записи без профиля), который работает только при `quality_eval_enabled=true` (новый setting, по умолчанию **выключен** — чтобы не было неожиданных трат на токены).
  - **API `/api/quality`:** rubrics CRUD (`GET/POST/GET/PATCH/DELETE /rubrics`, мутации — owner/admin; `is_default` сбрасывает флаг у остальных рубрик ws), `GET /records/{task_id}/profile`, `POST /records/{task_id}/evaluate`. `templates` API теперь принимает/отдаёт `rubric_id` (вкл. версионирование/rollback).

Сборка `npm run build` зелёная.
  - **Тесты:** `tests/unit/test_rubric.py` (8 — приоритет резолва, форма профиля, gate, deferred, изоляция ошибок, skip без модели) + `tests/integration/test_quality_eval.py` (6 — сидинг дефолтов, CRUD+единый default, workspace-изоляция, on-demand eval, skip без модели, scheduler уважает `quality_eval_enabled`). Весь backend-сьют: **201 passed**.
  - **Сознательные решения:** реализован только `judge`-evaluator (LLM-as-judge, O2); `objective` (E-04) и `human` (E-05) распознаются схемой, но в скоринге `deferred`. - **E-01 — Quality Data Lake.** Систематический сбор каждого выполнения задачи в неизменяемую versioned-запись для долгосрочного анализа/research/экспорта. Фундамент Phase 0 (`SPAWNHIVE_EVAL_BACKLOG.md`, Linear SPA-5).
  - **Новая таблица `quality_records`** (миграция `b1c2d3e4f5a6`): один снапшот на задачу (UNIQUE `task_id`), индексируемое summary в Postgres + полный JSON-блоб в MinIO (`data-lake/<ws>/<task>.json`). Nullable JSONB-слоты под downstream: `quality_profile` (E-02), `trajectory_profile` (E-07), `human_feedback` (E-05), `longitudinal` (E-22), `reproducibility` (E-20) — здесь оставлены NULL.

  - **Сборщик `app/quality/data_lake.py`:** `assemble_record` (из tasks + agent_events + agent_log_chunks + decomposition tree) и идемпотентный `build_quality_record` (summary-row + MinIO-блоб; повтор только реконсилит final_status/cost/tokens). Новые settings: `data_lake_retention_days` (0 = вечно), `data_lake_public_opt_in_default` (false).
