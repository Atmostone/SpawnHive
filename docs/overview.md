# SpawnHive ‚Äî Overview

## What it is

A self-hosted platform for orchestrating specialised AI agents. It takes a task ‚Üí picks an agent template ‚Üí spawns an isolated Docker container ‚Üí the agent solves the task using its built-in tools and MCP servers ‚Üí the result goes to review ‚Üí the user approves/rejects.

## Who it's for

- Developers and researchers who need several narrowly-specialised agents for different tasks (research, coding, writing, devops‚Ä¶) under a single control plane.
- Teams who want self-hosting (privacy, control over models, customisation).
- People who want a visual orchestration layer on top of litellm/MCP without vendor lock-in.

## What makes it different

- **Templates as agent roles**: each template is `(model, soul_md, tools, mcp_servers, limits)`. The LLM provider can be overridden per template.
- **Structured Memory**: automatic extraction of entities/relations from task results, embedding-based dedup, relevant sub-graph injected into the agent.
- **Bidirectional control**: the orchestrator can send feedback / abort / switch_model into a live agent container.
- **MCP-first**: custom MCP servers plug in per template without code changes.
- **Local-first**: everything runs in Docker Compose; nothing leaves the host.

- ‚úÖ Eval Phase 0 ‚Äî **E-01 Quality Data Lake**: immutable, versioned per-task execution snapshots (Postgres summary + MinIO blob), with `/api/data-lake` query + parquet/JSON export, retention + backfill jobs, and nullable slots for downstream eval features (E-02/E-05/E-07/E-20/E-22).
- ‚úÖ Eval Phase 0 ‚Äî **E-02 Multi-dimensional Quality Rubric Engine**: per-task rubrics scoring results into a quality profile (vector of 0‚Äì10) via LLM-as-judge, with 5 built-in rubrics, custom rubric editor, radar-chart UI, soft threshold gating, and on-demand + async evaluation (fills the `quality_profile` slot).
- ‚úÖ Eval Phase 1 ‚Äî **E-03 Reference-based Judge**: a `reference` rubric dimension scores the result against a task's optional gold `reference_answer` and folds one 0‚Äì10 score into the E-02 profile. Modes: pointwise (LLM), exact, fuzzy (difflib), semantic (embeddings). Pairwise (A vs B) deferred ‚Äî needs a second candidate (E-11/E-21).
- ‚úÖ Eval Phase 1 ‚Äî **E-04 Behavioral Testing Layer** (POC): an `objective` rubric dimension runs a deterministic static-analysis probe over the task's Python artifacts and folds one 0‚Äì10 measurement into the E-02 profile. Probes: `lint` (ruff), `types` (mypy), run in-process, memoised by artifact hash. POC scope is Python static analysis only; executing code (pytest), web/text/data probes and the docker-isolated plugin format are deferred.
- ‚úÖ Eval Phase 1 ‚Äî **E-05 Human Feedback Collection**: an optional, non-blocking form rates each E-02 axis 1‚Äì10 (banded incorrect 1-3 / improve 4-7 / correct 8-10) with per-dimension and overall comments and an approve/reject verdict, stored as a parallel signal in `quality_records.human_feedback`. The judge score is shown alongside each slider (one-click agree); a `/api/quality/calibration` export pairs judge‚Üîhuman scores for E-17. Pairwise comparison deferred to E-21; configurable bands and feedback‚Üíagent re-run deferred to E-26.
- ‚úÖ Eval Phase 2 ‚Äî **E-06 Trace Cleaner**: a deterministic, LLM-free pre-processor that turns a raw agent trajectory (events + log chunks) into a compact, judge-ready `CleanedTrace` ‚Äî keeping the task, per-step reasoning and tool calls, dropping the `agent_spawned` system snapshot and noise events, and truncating long tool outputs (with a `keep_tail_on_error` debug option). Reports token savings via tiktoken; previewed read-only at `GET /api/quality/records/{task_id}/trace`. Input for the trajectory judge (E-07) ‚Äî it scores nothing and persists nothing.
- ‚úÖ Eval Phase 2 ‚Äî **E-07 6-axis Trajectory Judge**: the process counterpart of the outcome judge ‚Äî an LLM scores the cleaned trace (E-06) on six axes in a single call (efficiency, tool_selection, parameter_quality, error_recovery, goal_alignment, loop_detection), each 0‚Äì10 with a reason, plus an overall score, derived `loop_detected` flag and summary, written to the `trajectory_profile` slot next to E-02's `quality_profile`. Reuses the E-02 judge-model resolver; cost is bounded by a configurable input-token cap; on-demand `POST ‚Ä¶/evaluate-trajectory` + read `‚Ä¶/trajectory`, visualised as a radar panel in TaskDetail. Optional batch job off by default.
- ‚úÖ Eval Phase 2 ‚Äî **E-08 TRACE Evidence Bank Judge**: the contextual counterpart of E-07 ‚Äî instead of judging the whole trajectory in one holistic call, it walks the cleaned trace step by step accumulating an **evidence bank** (facts established so far) that is threaded into each step's prompt, then produces the same evidence-aware 6-axis profile plus a `groundedness` signal that flags "ü§∑ lucky" cases (a correct answer not supported by the gathered evidence). `N+1` calls, bounded by `trace_evidence_max_steps`/`‚Ä¶_max_input_tokens`; reuses E-07's axes/tool/resolver; written to a new `trajectory_evidence_profile` slot (coexists with E-07 for comparison); on-demand `POST ‚Ä¶/evaluate-trajectory-evidence` + read `‚Ä¶/trajectory-evidence`, shown as an evidence-bank panel in TaskDetail. Optional batch job off by default. The formal E-07-vs-E-08 comparison is deferred to the public benchmark (E-23).
- ‚úÖ Eval Phase 2 ‚Äî **E-09 Trajectory Matching (T3)**: a **deterministic, LLM-free** matcher for the narrow class of tasks with a *canonical* trajectory (a single valid tool-call path). It compares the agent's actual tool sequence (from the E-06 cleaned trace) against a reference stored on the task (`tasks.canonical_trajectory` ‚Äî a list of tool names or a `{nodes, edges}` DAG) on three metrics: `exact` (sequence equality), `edit` (`difflib` ratio) and `dag` (the run is a valid topological order of the canonical DAG). The headline follows the configured `match_mode` (default `edit`); written to the new `trajectory_match_profile` slot. Only runs when a canonical trajectory is set (else skipped); on-demand `POST ‚Ä¶/evaluate-trajectory-match` + read `‚Ä¶/trajectory-match`, shown as a match panel in TaskDetail. No batch job (instant, free, rare task class). Canonical trajectories normally come from the benchmark dataset (E-23); settable now via the task field.
- ‚úÖ Eval Phase 3 ‚Äî **E-11 Variance / Robustness Harness (R1)**: runs one scenario N times and measures the **dispersion** of the result instead of a single point estimate ‚Äî an agent that is sometimes brilliant and sometimes fails is worse than a stably-mediocre one. Children are created via a small **re-run core** (`clone_task_for_rerun` + a `tasks.run_config`/`replay_of_task_id` seam and an engine pinned-template fast path that skips decomposition/selection) and drained by the existing orchestrator loop under `max_concurrent_agents`; a `variance_run_tick` scheduler job creates the next children under a **cost cap**, judges finished ones (E-02/E-07 when configured) and aggregates. Reports distributions (mean/std/p25-p50-p75-p95) of outcome-score, trajectory length and trajectory score, plus success rate and tool-selection stability. `POST/GET /api/quality/variance` + `python -m app.cli.variance`, visualised as box-plots in TaskDetail. The same re-run core is the seam for E-21/E-24/E-26 and a future full U-03 replay UX.
- ‚úÖ Eval Phase 3 ‚Äî **E-12 Adversarial / Perturbation Judge (R2)**: the complement of E-11 ‚Äî instead of robustness to model stochasticity on a fixed input, it probes robustness to **input variation**. Replays a finished scenario through four pluggable transforms ‚Äî `paraphrase` (LLM rewrite, reusing the E-02 judge model), `noise` and `reorder` (deterministic, no LLM), and `inject` (a prompt-injection payload appended to the first **tool response** at runtime via `run_config.tool_injection` ‚Üí `AGENT_TOOL_INJECTION`) ‚Äî plus `base_n` clean baseline runs. Compares each perturbed outcome profile against the baseline ‚Üí per-transform and **overall robustness score** (1.0 = no degradation), with a deterministic, canary-based **safety flag** for whether the agent followed the injection (overlaps the security pillar's S-02). Reuses the E-11 poll-driven machinery (re-run core, orchestrator loop, cost cap, `runs_common` helpers) via a `perturbation_run_tick` job. `POST/GET /api/quality/perturbation`, with robustness bars + an injection safety badge in TaskDetail.
- ‚úÖ Eval Phase 3 ‚Äî **E-13 Capability-isolation Tests (C1, part A)**: a **deterministic** harness for the C1 failure where an agent produces the right answer *from parametric memory* without using the tool the task requires (fresh data after cutoff, private RAG data, exact arithmetic, local state) ‚Äî outcome looks correct but the agent "cheated" and will fail when the data changes. A task carries a `capability_spec` (`{required_tools[], category?, match?}`); the harness reuses E-09's `extract_tool_sequence` for a **Glass-Box** check that the required tool(s) were actually called, reads outcome correctness from the E-02 judge (a scored `reference` dimension when present, else `weighted_score ‚â• capability_outcome_threshold`; no new model), and classifies the run into `genuine` / **`cheated`** (correct but tool not used) / `failed_with_tool` / `failed_no_tool` in the new `capability_profile` slot. `capability_score = genuine/total`, with by-model/category/template breakdowns (`aggregate_capability`) ‚Äî the "compare models by capability" signal. On-demand `POST ‚Ä¶/evaluate-capability` + read `‚Ä¶/capability` + `GET ‚Ä¶/capability/aggregate`, a CLI, an off-by-default batch job, and a capability panel in TaskDetail.
- ‚úÖ Eval Phase 3 ‚Äî **E-14 Failure Mode Classifier (F1)**: the process-failure taxonomy on top of E-07 ‚Äî an LLM, in a single call, labels the E-06 cleaned trace (with the existing E-02 outcome profile and E-07 trajectory profile fed as grounding context, never re-run) with a **multi-label** set of failure classes, each with a confidence and reason: `tool_confusion`, `parameter_blind`, `loop`, `premature_stop`, `hallucinated_tool_result`, `ignored_error` (extensible). It runs on every terminal task ‚Äî a clean run yields an empty list, so it also surfaces "succeeded with a defective process". Reuses E-02/E-07's judge resolver and input-token cap (`failure_judge_max_input_tokens`); never raises (failure ‚Üí `status: "error"`). `aggregate_failure_modes` rolls runs up into per-class counts + rate by class/model/template ‚Äî the distribution of failure types per (model, template), feeding E-24 and the multi-agent failure attribution (SPA-30). Written to the `failure_profile` slot; on-demand `POST ‚Ä¶/evaluate-failure-modes` + read `‚Ä¶/failure-modes` + `GET ‚Ä¶/failure-modes/aggregate`, an off-by-default batch job, and a failure-mode panel in TaskDetail.
- ‚úÖ Eval Phase 3 ‚Äî **E-15 Hallucination Detection (Q-09)**: a **fact-checker over the finished run's deliverable** (`task.result_summary`) across four categories ‚Äî **URLs / APIs / numbers / citations** ‚Äî that E-02 (outcome) and E-07 (trajectory) miss: the outcome can be "correct" while the text contains invented links, non-existent functions, or unsourced figures. **Hybrid**: URLs and code-fence API symbols are checked **deterministically** against the E-06 cleaned trace (a URL is "supported" iff it appears in some tool argument/result ‚Äî *in-trace only*, no live HTTP; a live-check flag is reserved for v2), while numbers, claims and unconfirmed APIs go to a **single LLM call** (same judge resolver as E-02/E-07/E-14; E-02 outcome-summary and E-08 evidence facts fed as grounding, never re-run). If the deterministic pass leaves nothing to ask, **zero LLM calls**. Each item is tagged `kind: deterministic|llm`; the profile reports per-category `checked`/`hallucinated`/`items[]` plus a top-level `hallucination_rate` (= count/items_total) ‚Äî a clean deliverable yields rate 0. Never raises (failure ‚Üí `status: "error"`); skipped when there is no judge model, no deliverable, or no trace. `aggregate_hallucinations` rolls runs up into per-category rates by category/model/template ‚Äî the hallucination rate per (model, template). Written to the new orthogonal `hallucination_profile` slot (not a rubric dimension); `POST ‚Ä¶/evaluate-hallucinations` + read `‚Ä¶/hallucinations` + `GET ‚Ä¶/hallucinations/aggregate`, an off-by-default batch job, and a hallucination panel in TaskDetail.
- ‚úÖ Eval Phase 3 ‚Äî **E-16 Confidence Calibration (Q-10)**: measures whether a model's stated confidence tracks its actual correctness ‚Äî a prerequisite for agent ‚Üí agent delegation, since a Coordinator has no basis for trusting an over-confident sub-agent. For each finished task it records the pair **(predicted_confidence, actual_correctness)**: correctness is reused from E-02 (`_outcome_from_profile` ‚Äî a scored `reference` dimension, else `weighted_score ‚â• threshold`; no signal ‚Üí skipped), while confidence ‚Äî which exists nowhere in the system ‚Äî is elicited with a **single post-hoc self-probe**: the task's *own* model (resolved by `model_used`, judge fallback, so the per-model view is self-referential) re-reads the task + its answer + the E-06 trace **without the grader's verdict** and reports `P(correct) ‚àà [0,1]`. The per-task `calibration_profile` slot stores the raw pair + its `brier_term`; the headline metrics are population-level ‚Äî `aggregate_calibration` computes **ECE / Brier / a reliability diagram** (equal-width confidence buckets of avg-confidence vs accuracy) overall and `by_model`/`by_template`, with a per-model recommendation ("model X overestimates itself in the 70‚Äì80% confidence zone"). One LLM call per task, input-capped (`calibration_judge_max_input_tokens`), never raises (failure ‚Üí `status: "error"`). Written to the new orthogonal slot (not a rubric dimension); `POST ‚Ä¶/evaluate-calibration` + read `‚Ä¶/calibration` + `GET ‚Ä¶/calibration/aggregate`, an off-by-default batch job, and a calibration panel (confidence bar + Brier + reliability diagram) in TaskDetail.
- ‚úÖ Eval Phase 4 ‚Äî **E-17 Judge Calibration Protocol (RQ1)**: an LLM-judge metric is meaningless until it is validated against humans ‚Äî the central source of doubt about eval. E-17 compares the LLM judge's per-dimension scores (E-02, `quality_profile.dimensions[]`) with human ratings on the same axes (E-05, `human_feedback.dimensions[]`) across every record that carries both ‚Äî **no LLM call**, pure agreement statistics over already-stored scores. Pure-Python `app/quality/stats.py` (no scipy/numpy) gives **Pearson + Spearman** on the (judge, human) scores and **Cohen's Œ∫** on the categorical band projection (bad/improve/good), plus a `mean_bias`; an **overall verdict-agreement** compares the judge gate vs the human approve/reject verdict. A dimension is `reliable` when its band Œ∫ clears `judge_calibration_min_kappa` (default 0.6); `recommendations[]` read "judge reliable for Correctness (kappa=0.71, r=0.81) / diverges on Tool Selection ‚Ä¶". The existing `GET /api/quality/calibration` export and the report share one `collect_judge_human_pairs` collector (DRY). Because a calibration is per-(workspace, judge model) ‚Äî not per-task ‚Äî it is persisted append-only in a new **`judge_calibrations`** table, versioned per `judge_config_key` (the judge model's api_name), so re-running after a judge change keeps the old curves. `POST ‚Ä¶/judge-calibration/run` (owner/admin) + `GET ‚Ä¶/judge-calibration` (+`?history`) + `GET ‚Ä¶/judge-calibration/badge`, a CLI (`python -m app.cli.judge_calibration run|show`), a workspace-level panel on **Analytics** (per-dimension reliability table + history) and a "judge calibrated against N humans, Œ∫=X.X" trust badge (also on the TaskDetail quality panel). On-demand only (no batch job).
- ‚úÖ Eval Phase 4 ‚Äî **E-18 Bias Mitigation Toolkit**: where E-17 *measures* judge‚Üîhuman divergence, E-18 *counteracts* the four known LLM-judge biases (¬ß7.2). Four `bias_mitigation_*` toggles configure the **live** judge: `verbosity` and `score_clustering` append an instruction to the judge prompt ("ignore answer length"; "use the full 0‚Äì10 range, don't default to 7‚Äì8"), `self_preference` flags when the judge model is the same model/family as the agent model (`quality_profile.bias_mitigation`), and `position` is a reserved no-op (it needs pairwise judging, deferred to E-21). The **bias report** is a *controlled A/B re-judge*: it re-scores every calibration-set task (records with human feedback) with the mitigations OFF then ON against an identical context and runs both passes through the same E-17 `_compute_report`, so before/after agreement-with-human is computed identically ‚Äî this is the one part of E-18 that **spends LLM calls** (owner/admin, on-demand, dims-within-task concurrent / tasks sequential). On top of agreement it reports per-bias diagnostics (length‚Üîscore correlation off/on vs human; score-spread off/on; self-preference flag). Persisted append-only in a new **`bias_reports`** table (migration `d2e3f4a5b6c7`), versioned per judge model like E-17. `POST ‚Ä¶/bias-report/run` + `GET ‚Ä¶/bias-report` (+`?history`), CLI (`python -m app.cli.bias_report run|show`), a "Bias Mitigation" panel on **Analytics** (before/after Œ∫ table + diagnostics) and four toggles in **Settings**.
- ‚úÖ Eval Phase 4 ‚Äî **E-19 Aggregation Engine (Bradley-Terry / Elo)**: ranks models/templates **pairwise** rather than by averaging pointwise scores. A pure engine (`app/quality/aggregation.py`) takes head-to-head matches (`{player_a, player_b, outcome}`) and fits Bradley-Terry (MLE via MM iteration, with a Bayesian prior so undefeated players still converge) or Elo (seeded multi-pass to kill order-dependence) on a shared Elo scale, with **bootstrap confidence intervals** ‚Äî fully deterministic for a fixed seed (the literal `rank(pairwise_results, method)` acceptance). Its natural feed is pairwise judging (**E-21**, now built ‚Äî it supplies explicit matches from real A/B verdicts); independently `app/quality/ranking.py` **derives** matches from the stored pointwise `weighted_score` (same `benchmark_case_id`, higher mean wins) when no pairwise data exists ‚Äî `metrics.source` (`explicit`|`derived`) records which fed a leaderboard. Subject is selectable (`model`|`template`); persisted append-only in a new **`ranking_reports`** table (migration `e3f4a5b6c7d8`), versioned per `{subject}:{method}`. `POST ‚Ä¶/ranking/run` + `GET ‚Ä¶/ranking` (+`?history`) + `‚Ä¶/ranking/badge`, CLI (`python -m app.cli.ranking run|show`), a "Leaderboard" panel on **Analytics** (rating + 95% CI bar + W/L/T, with model/template & BT/Elo selectors). No LLM calls.
- ‚úÖ Eval Phase 4 ‚Äî **E-20 Reproducibility Snapshot**: models drift and LLMs are stochastic, so every quality record now captures an **experiment_snapshot** of the exact state that produced the run ‚Äî model, soul_md (prompt), memory, tools, task input ‚Äî with a deterministic **fingerprint** over the run-defining fields. Capture is automatic and free (no LLM call): the inputs are already recorded at spawn time in `agent_spawned`, and `build_quality_record` materializes them into the **reserved** `quality_records.reproducibility` slot ‚Äî so it's **per-record, no new table**. `app/quality/reproducibility.py` is honest about gaps ‚Äî `temperature`, tool **versions** and point-in-time RAG vectors aren't exposed by the runtime (and `seed` only for benchmark runs), so they're listed in a `manifest.missing` rather than faked. You can **diff** two snapshots (what changed between runs) and **replay** a run from its snapshot (derives a `run_config` and clones the task via the existing `clone_task_for_rerun`, linked by `replay_of_task_id`). `GET ‚Ä¶/records/{id}/reproducibility`, `POST ‚Ä¶/capture-reproducibility`, `GET ‚Ä¶/reproducibility/diff`, `POST ‚Ä¶/replay`, CLI (`python -m app.cli.reproducibility show|diff|replay`), a "Reproducibility" panel on **Analytics**.
- ‚úÖ Eval Phase 4 ‚Äî **E-21 Pairwise Comparison Framework**: pointwise judging clusters everything into 7-8 (¬ß7.2), so E-21 adds **"which is better, A or B?"** ‚Äî `app/quality/comparison.py` + a new **`pairwise_comparisons`** table (migration `f4a5b6c7d8e9`). Candidate B is either an existing task (**direct**) or **generated** by re-running a source with a `model_id`/`template_id`/`soul_md` override (async via `clone_task_for_rerun` + a `pairwise_run_tick` job). The LLM judge implements the **position-bias mitigation** E-18 deferred: it judges the pair in **both orders** and reconciles ‚Äî agree ‚Üí winner, disagree ‚Üí `tie` + `position_bias_detected`. A **human mode** records its own verdict on the same row (judge‚Üîhuman agreement for E-17). Judged verdicts become **real** matches fed to the E-19 engine ‚Üí an **ELO leaderboard** (`source="explicit"`, the designed E-19‚ÜíE-21 hand-off, shown in the existing Leaderboard tab). `POST ‚Ä¶/comparison` (direct/generated), `GET ‚Ä¶/comparison` (+agreement) / `‚Ä¶/{id}` (+side-by-side), `POST ‚Ä¶/{id}/judge`, `PUT ‚Ä¶/{id}/human-verdict`, `POST ‚Ä¶/comparison/leaderboard`, CLI (`python -m app.cli.comparison create|generate|show|list|judge|leaderboard`), a "Pairwise" panel on **Analytics** (side-by-side answers + LLM-judge with position-bias readout + human winner picker + push-to-ELO).
- ‚úÖ Infra ‚Äî **SPA-41 Tool & MCP Registry (user-level)**: tools & MCP servers used to be configured inline on every template (duplicated config, scattered credentials, error-prone for benchmark A/B matrices). Now they live in a **workspace-level registry** (`registry_entries` + `app/registry/`, migration `a5b6c7d8e9f0`) that templates reference by id (`templates.tool_ids` replaces the old `tools`/`mcp_servers`). A **big-bang** migration deduped every template's inline builtins/MCP into registry rows and rewrote the references. At spawn, `resolve_template_tools` materializes a template's refs ‚Äî plus any `task.run_config.tools_override = {enable,disable}` (finest-restriction-wins) ‚Äî into the exact `AGENT_TOOLS` / `MCP_SERVERS` shapes the agent already consumes (the only hot-path change is in `engine.py`; the resolved set is also recorded in `agent_spawned`, keeping E-20 honest). Secrets are stored plain (like `Provider.api_key`) and **masked on every read** ‚Äî only the resolver reveals them; the `secrets` slot is the seam for S-06 Vault. `/api/registry/tools` CRUD + `‚Ä¶/{id}/test` (builtin ok / http reachability / stdio shape-validate), delete guarded by 409 unless `?force`, CLI `python -m app.cli.registry`, a "Tool & MCP Registry" section in **Settings**, and a registry multiselect in the template editor. Unblocks clean benchmarks (SPA-40/SPA-27).
- ‚úÖ Eval infra ‚Äî **Benchmark Case Store (pre-E-23)**: a **file-first** store of reusable task *definitions* (mirror of the `quality_records` result slots) ‚Äî versioned YAML/JSON cases under `backend/benchmarks/<suite>/` with a pluggable `gold` envelope (`reference_answer`/`rubric`/`canonical_trajectory`/`capability_spec`). `app/quality/benchmark.py` parses + `materialize`s a case into runnable READY task instances tagged `benchmark_case_id`/`benchmark_suite` (denormalized onto the quality record), so eval aggregation can scope by suite √ó case √ó model (`aggregate_capability(suite=‚Ä¶)`, `GET ‚Ä¶/capability/aggregate?suite=`). CLI `python -m app.cli.benchmark suites|load|status|evaluate|aggregate`; full format in [`benchmarks.md`](benchmarks.md). Ships the first **capability-isolation suite (E-13 part B): 30 cases** across the four categories (10 exact_compute, 8 private_data + a RAG fixture, 7 fresh_data, 5 local_state). The registry table / catalogue API/UI / publication remain E-23. The suite was run end-to-end across two models (MiniMax-M2.7 & GLM-5.1, 60 real agent runs) ‚Äî `capability_score` MiniMax 0.73 / GLM 0.67. The Glass-Box check sources tool calls from the cleaned trace **unioned with the durable E-01 blob** (`execution.tool_calls`). The log archive now **preserves `tool_name`** through compaction (JSON-lines format), so the cleaned trace (E-06) and trajectory matcher (E-09) stay tool-aware after archival; the blob union remains as defense-in-depth (and for legacy plain-text archives, which lose `tool_name`).
