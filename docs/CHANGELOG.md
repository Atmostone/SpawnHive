## 2026-06-11

- **SPA-40 — Experiment Runner / A/B Matrix Harness.** First-class объект **Experiment**: замороженный датасет × матрица конфигураций × `n_runs_per_cell` → раннер → отчёт. Превращает eval-стек (E-01…E-21) в один пользовательский инструмент: «MiniMax vs DeepSeek», «улучшает ли v2 промпт», «нужен ли оркестратор» — одна операция со статистикой.
  - **Benchmark execution path**: дети эксперимента несут `run_config.benchmark_mode` + `origin='experiment'` + `max_retries=0`; вебхук ведёт `completed` сразу в DONE (без inline-ревью, approval и ретраев — они искажали бы N-run семантику), quality record строится и для DONE. Доска скрывает `origin='experiment'` (новая колонка `tasks.origin`, `GET /api/tasks?include_experiments=`).
  - **`orchestrator: on|off` — ось матрицы**: off → пин `template_id` (fast path движка, без декомпозиции/выбора шаблона); on → полная оркестрация, дети декомпозиции наследуют benchmark run_config + origin, корень получает rollup-резюме для E-02. Отдельный view «с оркестратором vs без» в отчёте.
  - **Оси конфигурации**: template, model, temperature/seed (теперь реально доезжают до агента: `LLM_TEMPERATURE`/`LLM_SEED` в env + чтение в `agent-image/agent.py`), soul_md, tools_override (ссылки на реестр SPA-41 — честное A/B), memory_mode. Композиция: явный список И/ИЛИ декартово произведение `axes`; дедуп по канон-фингерпринту; лимиты 24 конфига / 300 кейсов / 1000 ранов.
  - **Датасет замораживается** при создании (suite / существующие задачи / upload JSONL с понятной построчной валидацией) в `experiments.dataset_cases`; дети тегируются `benchmark_suite="exp:<id>"` — вся E-01 обвязка работает без изменений.
  - **Per-case rubric**: кейс может нести inline-рубрику (`{name?, dimensions:[...]}` — валидируется при upload, pydantic `UploadRubric`); при settle она **переопределяет** рубрику шаблона/воркспейса для E-02 (`evaluate_task_quality(rubric_override=...)`, `rubric_id: null` в профиле). Смешанные датасеты (математика + письмо в одном эксперименте) судятся правильными измерениями по-кейсово, а не одной рубрикой шаблона.
  - **Раннер** — poll-driven tick `experiment_run_tick` (20с, паттерн E-11): ячейки пре-создаются `pending` в `experiment_runs` (идемпотентный клейм, точный прогресс, результаты переживают удаление задач — скоры/стоимость денормализуются), бюджет-кап → `skipped`+статус `capped` с частичными результатами, pause/resume/cancel.
  - **Отчёт** (кэшируется по завершении): summary по конфигам, heatmap (конфиги × измерения рубрики), Парето-фронт (качество×цена×время), scatter outcome×trajectory, pairwise-лидерборд (E-19 BT/Elo + bootstrap CI из pointwise-скоров, спаренных по кейсам), **статзначимость** (Welch t-test с точным t-CDF на чистом Python + Mann-Whitney U, ★ p<0.05), failure modes, сравнение on/off.
  - **API** `/api/experiments` (CRUD/preview/run/pause/resume/cancel/report/results/clone/export CSV+JSON), **UI** `/experiments` (форма с live-preview стоимости, прогресс-матрица, вкладки отчёта, Clone/Re-run), **CLI** `python -m app.cli.experiment`. Миграция `b6c7d8e9f0a1` (experiments, experiment_runs, tasks.origin), round-trip. Тесты: +65 (юниты на статистику против табличных значений, матрицу, датасеты, отчёт; интеграционные на webhook-bypass, lifecycle, API); полный прогон — 620 passed + тот же 1 pre-existing fail (`test_human_feedback`).

- **E-21 — Pairwise Comparison Framework.** Точечное судейство (E-02) скучивает всё в 7-8 (§7.2); попарное — «что лучше, A или B?» — надёжнее и естественнее для человека (Phase 4, Linear SPA-25; разблокирует фид E-19). Acceptance: side-by-side UI на вкладке Analytics; LLM-судья с **митигацией position-bias** (E-18); **human-режим** (для калибровки E-17); **ELO-leaderboard** через E-19.
  - **Новая таблица `pairwise_comparisons`** (миграция `f4a5b6c7d8e9`, round-trip): одна строка — это head-to-head «A vs B» по оси `subject` (`model`/`template`/`prompt`). Оба вердикта (судья + человек) живут на строке → agreement судья↔человек локален (линковка E-17). Кандидат B — либо готовая задача (**direct**, `status=ready`), либо **генерируется** реран'ом источника с оверрайдом (`model_id`/`template_id`/`soul_md`) — async через `clone_task_for_rerun` + scheduler-tick `pairwise_run_tick` (зеркало variance/perturbation, без setting-гейта).
  - **Митигация position-bias (то, что E-18 отложил как no-op).** `judge_pair_llm` судит одну и ту же пару в **обоих порядках** — `(A,B)` и `(B,A)` — и сверяет: совпали → победитель; разошлись → `tie` + `position_bias_detected=true`. Per-order вердикты пишутся в `judge_detail`. Два LLM-вызова на пару; `mitigate_position=False` (один вызов) оставлен, чтобы *измерять* сырой bias. Forced tool-call `choose_winner`, модель судьи через `_resolve_judge_model` (как E-02).
  - **ELO через E-19 — задуманная передача.** `run_pairwise_leaderboard` собирает judged-сравнения, `comparisons_to_matches` превращает вердикты в реальные матчи (`{player_a, player_b, outcome, weight}`, self/incomplete/без-вердикта отброшены) и вызывает `run_ranking(matches=…, method="elo")` → версионируемый `ranking_report` (`source="explicit"`) в **существующей** вкладке Leaderboard. Тот же `ranking_key="model:elo"`, что у derived-пути E-19 — различаются по `metrics.source` (`explicit` vs `derived`); это и есть hand-off E-19 → E-21, не новый ключ.

- **E-19 — Aggregation Engine (Bradley-Terry / Elo).** Точечные оценки (E-02) дают одно число на задачу; устойчивее ранжировать **попарно** — много «A vs B» матчей, агрегированных в глобальный рейтинг с доверительным интервалом (Phase 4, Linear SPA-23; формально зависит от E-21, который ещё не построен — это зависимость по *данным*, не по коду). Acceptance: `rank(pairwise_results, method='bt'|'elo')` → ранжированный список с CI + leaderboard-визуализация.
  - **Чистый движок** (`app/quality/aggregation.py`, без БД и LLM, тривиально юнит-тестируемый и переиспользуемый будущим E-21). Матч — `{player_a, player_b, outcome:"a"|"b"|"tie", weight}`. `bradley_terry` — MLE-силы через классическую MM-итерацию с малым байесовским **prior** (виртуальные победа+поражение против фантомного среднего соперника), чтобы непобеждённый игрок или несвязный граф сравнений всё равно сходились к конечному рейтингу; `elo` усредняет последовательные апдейты по нескольким seeded-перемешанным проходам, убирая зависимость от порядка. Оба метода мапятся на общую **Elo-шкалу** (центр 1500). CI — seeded bootstrap (ресэмпл матчей с возвратом, переоценка, перцентильная полоса p2.5/p97.5; меньше матчей → шире CI). Вся случайность через `random.Random(seed)` → полностью детерминировано (запинено тестом).
  - **Источник матчей.** Естественный фид E-19 — pairwise-судейство (**E-21**), которого ещё нет. Пока его нет, `app/quality/ranking.py::derive_matches_from_records` строит мост: превращает уже сохранённые точечные `quality_profile.weighted_score` в матчи — внутри одного `benchmark_case_id` модель/шаблон с большим средним скором «побеждает» (разрыв ≤ `ranking_tie_epsilon` → ничья). Чистое спаривание (`build_matches`) тестируется отдельно. Можно также передать **явный** список матчей (буквальный `rank(pairwise_results)` API), минуя деривацию. Когда E-21 появится — он просто отдаёт настоящие матчи в тот же `run_ranking`.
  - **Subject** выбирается: ранжируем `model` (`quality_record.model_used`) или `template` (`template_name`). Тюнинги (`ranking_tie_epsilon`=0.5, `ranking_bootstrap_resamples`=200, `ranking_seed`=0) — в free-form **settings**, без миграции.
  - **Данные:** новая таблица **`ranking_reports`** (миграция `e3f4a5b6c7d8`, down_revision `d2e3f4a5b6c7`) — зеркало `judge_calibrations`/`bias_reports`: append-only, версионируется per `(workspace, ranking_key)` где `ranking_key="{subject}:{method}"`, полный leaderboard в `metrics` (`players[]` = `{player, rating, ci_low, ci_high, rank, wins, losses, ties, n_matches, win_rate}`). `passed` = leaderboard построен (`status=="ok"`).
